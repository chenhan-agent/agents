# 中文 Emoji 表意設計評分規則

這份 rubric 用來評估：

1. **generator agent** 把 target 轉成 emoji 的設計品質
2. **solver agent** 能不能從 emoji 正確還原 target，且理由是否對得上

總分預設為 **10 分**。

## 評分重心

這份 rubric 雖然保留「答案正確度」，但在 benchmark 解讀上：

- **答案正確度** 是 baseline
- **理由對齊度** 是核心指標
- **理由品質** 與 **難點辨識** 用來看 solver 是否真的理解設計

如果是要看 **reasoning gap**，請優先看：

1. `理由對齊度`
2. `理由品質`
3. `難點辨識`
4. 最後才看 `答案正確度`

## A. 答案正確度（0-2）

- **2**：答案完全正確
- **1**：答案幾乎正確，或猜到同類型 / 同範圍但不夠精準
- **0**：完全錯誤或無法作答

## B. 理由對齊度（0-4）

看 solver agent 的理由是否對到 generator agent 原本設計的關鍵線索與轉折。

- **4**：抓到大多數核心線索、轉折與設計者真正想放的重點
- **3**：抓到多數核心線索，但漏掉部分轉折或文化層次
- **2**：抓到部分核心線索，但漏掉重要轉折
- **1**：只有零碎對到，整體推理不成形
- **0**：理由與出題設計幾乎無關

## C. 理由品質（0-3）

看 solver agent 的理由是否合理、有結構，而不是事後硬湊。

- **3**：推理清楚、有條理，能解釋每個 emoji 的作用，且層次分明
- **2**：大致合理，有結構，但深度不夠或有小跳躍
- **1**：部分合理，但有明顯跳躍或硬掰
- **0**：幾乎沒有真正推理

## D. 難點辨識（0-1）

看 solver agent 是否看出這組 emoji 的難點、誤導點或轉折點。

- **1**：有指出哪個線索最關鍵、哪裡容易誤判
- **0**：沒有辨識題目難點

## 分數解讀

- **9-10**：強通過
- **7-8**：通過
- **5-6**：部分通過
- **0-4**：失敗

## Judge checklist

judge agent 打分時要回答：

1. solver agent 的答案是否正確？
2. solver agent 是否抓到 generator agent 真正設計的線索？
3. solver agent 的理由是推理還是事後硬掰？
4. solver agent 是否有指出題目的關鍵轉折？

如果 benchmark 目標是 **reasoning gap**，judge 還要額外回答：

5. solver 是真的讀懂設計，還是只靠表面配對猜中？
6. 即使答案猜對，理由是否仍停在表層？

## Suggested judge output format

如果沒有另外指定格式，judge agent 預設輸出：

1. `答案正確度`
2. `理由對齊度`
3. `理由品質`
4. `難點辨識`
5. `總分`
6. `評語`
